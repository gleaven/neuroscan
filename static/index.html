ℹ How Abliteration Works ▼

What you'll learn: Whether this model's safety can be surgically removed — and what that reveals about the quality of its alignment.

What to do: Follow the 5 steps below. Each step unlocks the next.

  • Configure — Choose how many prompt pairs to analyze
  • Compute — Find the internal “refusal direction” the model uses to say no
  • Test — Try a prompt the model would normally refuse, with refusal removed
  • Optimize — Auto-tune parameters using Bayesian optimization
  • Export — Permanently modify weights or export the decensored model

Security insight: If abliteration easily removes refusal, the model’s safety alignment is superficial — a thin linear feature rather than deeply integrated behavior.

1 Configure Dataset & Layers ▶
What is this step doing?

Before we can remove a model’s ability to refuse harmful requests, we need training data: pairs of prompts where one is harmful (“How do I pick a lock?”) and one is harmless (“How do I bake cookies?”). By running both through the model and comparing the internal activations, we can find the direction in activation space that represents “I should refuse this.”

Dataset — The HuggingFace dataset provides 128 carefully curated pairs (recommended for accuracy). The built-in set has only 16 pairs but runs faster — good for a quick demo.

Samples — How many of those pairs to actually use. More samples = more reliable refusal direction, but takes longer. 128 is a good balance; 16 is fast for experimentation.

Activation Layers — A transformer model processes text through many layers, each building on the previous. The “refusal signal” might live in different parts of the internal pipeline:

  • resid_post (recommended) — The main information highway after each transformer block processes the data. This is where most safety-relevant signals accumulate, since it captures the combined effect of attention + MLP at each layer.
  • resid_pre — The information highway before each block. Comparing pre vs. post tells you what each block added.
  • attn_out — Just the attention mechanism’s contribution. Select this to check if refusal is driven by the model “looking at” certain tokens.
  • mlp_out — Just the feed-forward network’s contribution. Select this to check if refusal is stored as factual/associative knowledge.

Why select multiple? Checking multiple layer types simultaneously tells you where in the architecture the refusal signal is strongest. If attn_out shows a strong signal but mlp_out doesn’t, refusal is attention-driven. If all four show similar signals, the refusal behavior is deeply distributed. In practice, resid_post alone is sufficient for most abliteration work — the other options are for deeper investigation.

2 Compute Refusal Direction ▶
What is this step doing?

This is where the actual science happens. The system runs every harmful and harmless prompt from your dataset through the model, recording the internal activations (the numbers flowing through the neural network) at every layer.

It then computes the mean difference between harmful and harmless activations. This difference vector is the “refusal direction” — the internal signal the model uses to decide “I should refuse this request.”

Quality Metrics will appear after computation:

  • SNR (Signal-to-Noise Ratio) — How clearly the refusal signal stands out from background noise. Higher = cleaner signal, easier to remove. An SNR below 1.0 means the signal is buried in noise.
  • Quality Score — Overall confidence that we found a real refusal direction (0–1). Above 0.7 = strong signal, the model’s safety is “shallow” (linear, easy to remove). Below 0.3 = the model may have more robust alignment.
  • Explained Variance — How much of the harmful-vs-harmless difference is captured by this single direction. High variance = refusal is a simple, one-dimensional feature. Low = refusal is complex and multi-dimensional.

Save/Restore Cache — Computing directions takes time (running 256+ prompts through the model). The cache saves your computed directions so you can restore them instantly in a future session without recomputing.

3 Test Abliteration ▶
What is this step doing?

Now we test whether the refusal direction we found actually controls the model’s safety behavior. The system generates two responses to your prompt: one normal (with safety intact) and one abliterated (with the refusal direction subtracted from the activations at inference time).

If the normal response refuses but the abliterated response complies, abliteration successfully removed the model’s safety guardrail — proving the alignment was a linear, removable feature rather than deeply integrated behavior.

Method — How to subtract the refusal direction:

  • Standard — Simply projects out the refusal direction. Fast, but can distort the activation magnitudes, sometimes producing gibberish or repetitive text.
  • Norm-Preserving (recommended) — Subtracts the refusal direction but rescales the result to maintain the original activation magnitude. This preserves model coherence while still removing refusal. Best default choice.
  • Biprojected — Projects out the refusal direction from both sides of the weight matrices. Most expensive but produces the highest-quality outputs because it modifies the model’s geometry minimally.

KL Divergence — Measures how much the model’s output probability distribution changed after abliteration. Below 0.05 = the model is still smart. Above 0.2 = significant capability loss — try a different method.

Perplexity Check — Measures text fluency. If perplexity increases by more than 15%, the abliteration damaged the model’s language ability.

Concept Cones (Advanced) — Recent research (2025) shows that safety isn’t just one direction but a multi-dimensional “cone.” Computing multiple orthogonal directions via SVD lets you remove more of the safety behavior, at the cost of potentially more capability loss.

4 Auto-Optimize (Optuna TPE) ▶
What is this step doing?

In Step 3 you tested one set of parameters manually. But there are hundreds of possible combinations: different abliteration methods, different layer ranges, different weight strengths. Which combination gives the best jailbreak while keeping the model intelligent?

The optimizer uses Optuna TPE (Tree-structured Parzen Estimator) — a Bayesian optimization algorithm that intelligently explores the parameter space. Unlike grid search (trying every combination), TPE learns from each trial which regions of parameter space are promising, focusing its search where results are best.

It optimizes two objectives simultaneously:

  • Minimize refusal rate — What percentage of harmful prompts does the model still refuse? Lower = more successful abliteration.
  • Minimize KL divergence — How much did the model’s general capabilities degrade? Lower = smarter model post-abliteration.

These objectives often conflict — aggressively removing refusal tends to also remove intelligence. The result is a Pareto front: a curve of optimal trade-offs where no solution is better on both objectives. You pick the point on the curve that matches your needs.

Trials — How many parameter combinations to test. 30 is fast (~2-3 min); 100+ gives more thorough coverage but takes proportionally longer.

Test prompts — How many harmful prompts to test each trial against. More prompts = more reliable refusal rate estimate, but each trial takes longer.

Apply Best Parameters — Once optimization finishes, this button copies the best-found settings back to Step 3 so you can generate with them.

What is this step doing?

Steps 2–4 modified the model’s behavior at inference time — the refusal direction was subtracted from activations as they flowed through the network, but the actual model weights were never changed. This means the original model is still intact in memory.

Apply Permanently takes a fundamentally different approach: it modifies the model’s actual weight matrices by orthogonalizing out the refusal direction. This means the refusal behavior is mathematically removed from every MLP and attention layer’s weights. The model literally cannot refuse anymore — it’s not being overridden at runtime, the knowledge of how to refuse has been erased from the weights.

Why would you do this? For research: to study what a model looks like without safety, to test downstream defenses against an uncensored model, or to create a baseline for measuring alignment techniques.

Export Model — Saves the modified weights to disk as a standard HuggingFace model that can be loaded by any framework. This is how “uncensored” model variants are created in the open-source community.

Revert to Original — Discards all weight modifications and reloads the original model from disk. All computed directions are also cleared — you’d need to re-run Step 2 (or restore from cache).

Security implication: The ease of this process demonstrates why alignment through post-training safety tuning alone is fragile. If a single linear direction can be removed to bypass all safety, the model’s alignment was never deeply integrated into its reasoning.

ℹ How GCG Works ▼

What you'll learn: Whether the model can be forced to produce harmful output using optimized gibberish suffixes — and why this is a critical threat for anyone hosting an LLM.

What to do:

  1. Set a target — Enter the text you want to force the model to output
  2. Run the attack — The algorithm optimizes a nonsense suffix that hijacks the model. Watch the loss drop.
  3. Prove it works — Feed the suffix to the model and verify it generates your target
⚠ How Attackers Use GCG

GCG (Greedy Coordinate Gradient) is an automated adversarial attack from “Universal and Transferable Adversarial Attacks on Aligned Language Models” (Zou et al., 2023). It uses gradient-based optimization against the model’s own weights to construct token sequences that force specific outputs, even from safety-aligned models (RLHF, Constitutional AI, etc.).

Practical attack scenarios:

  • Safety bypass: Craft a suffix that makes the model produce content it was trained to refuse. The suffix looks like gibberish but operates directly in the embedding space.
  • Transfer attacks: Suffixes discovered on open-weight models can transfer to closed-source APIs. Run cheap optimization locally, then use the suffix against production endpoints.
  • Automated at scale: Unlike manual jailbreaks, GCG can be scripted to systematically probe every refusal category, generating thousands of bypass suffixes in hours.
  • Perplexity filter evasion: Newer variants add readability constraints, producing suffixes that look like plausible English rather than obvious token salad.
🛡 What This Means for Model Hosts

If you deploy an LLM — via API, chatbot, agent, or product — GCG demonstrates that alignment alone is not sufficient defense:

  • Safety training is a speed bump, not a wall. RLHF and refusal training add cost for attackers but can be systematically defeated with gradient access or transfer attacks.
  • Input filtering alone fails. GCG suffixes are semantically meaningless, so keyword/regex filters miss them. Perplexity filters help but can be evaded.
  • Defense in depth is required. Effective layers include: output monitoring, rate limiting, embedding-space anomaly detection, and application-level guardrails (e.g., NeMo Guardrails).
  • Continuous red-teaming is essential. New attack variants emerge regularly (AutoDAN, PAIR, TAP, Rainbow Teaming). Testing should be ongoing, not one-time.

This tool lets you measure your model’s attack surface before deployment. If the loss drops below ~0.5 within a few hundred steps, the model has weak adversarial robustness at that layer.

ℹ Unicode Steganography Attacks ▼

What you'll learn: Whether models process hidden instructions embedded in Unicode text that humans can’t see.

Attack methods:

  • Zero-Width Injection — Invisible Unicode characters (U+200B, U+200C, U+200D, U+FEFF) inserted between visible emoji/text. The model may “see” these while humans see nothing.
  • Tag Characters — Unicode Tags (U+E0001–U+E007F) encode hidden ASCII text within emoji sequences. Used in flag emoji (e.g., 🏴󠁧󠁢󠁥󠁮󠁧󠁿 = England flag). An attacker can embed arbitrary text in tag sequences.
  • Homoglyph Swap — Replace Latin characters with visually identical Unicode lookalikes (Cyrillic ‘a’ for Latin ‘a’) to bypass content filters while preserving readability.
  • Bidirectional Override — RLO/LRO characters that reverse text display order, making “mrah” appear as “harm” to the model while showing differently to humans.

These attacks test whether the model's tokenizer processes hidden Unicode the same way a human reader would — or whether invisible instructions “leak through” to influence the model.

ℹ How Benchmarks Work ▼

Standard academic benchmarks to evaluate model alignment:

  • TruthfulQA MC1 — 50 questions testing factual accuracy vs. common misconceptions. Score = % correct.
